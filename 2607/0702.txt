徐さんより頂いたレビュー指摘事項No.2「apq削除後のindex番号の変更要否」について、方針確認をお願いしたく連携いたします。

【現状の実装】
OperateTargetAPIKeys の apq（index: 4）を削除する際、後続の番号（5, 6, 7…）は詰め直さず、そのままの番号を維持しております。

【背景・懸念点】
本設定の index 値は、ALBのListener Rule Priority算出ロジックに使用されており、固定デュアルスタック化（A面/B面のインフラ固定化）が完了した後は、index が実際のALB Priorityと恒久的に紐付く想定です。
このため、「index を詰め直すかどうか」によって、今後の影響範囲が大きく変わってまいります。

【案①：番号を詰め直さない（現状の実装）】

メリット：削除対象以外のAPI Keyへの影響が一切なく、影響範囲を最小化できる
デメリット：index は「35以上設定不可」の上限があるため、欠番が発生し続けると将来的にindex資源を消費し枯渇するリスクがある

【案②：番号を詰め直す】

メリット：現時点（固定デュアルスタック化前）であれば、実際のALB Priorityへの影響はなく、index資源を無駄にしない
デメリット：固定デュアルスタック化完了後に同様の削除対応が発生した場合、番号詰め直しにより後続の全API KeyのALB Priorityが変更対象となり、優先度の重複エラーや、影響範囲が想定以上に拡大するリスクがある

【確認したい点】
今回のapq削除対応そのものだけでなく、今後の API Key 廃止対応における番号管理方針を、案①・案②のどちらに統一するか、ご定夺いただきたく存じます。
（固定デュアルスタック化完了前後で運用ルールを変える、というご判断も含め、ご意向をお伺いできればと思います。）
